# War Room Operations

War Room coordinates the campaign; child repositories own the code.

## Boundaries

- `repos.yaml` is the machine-readable map for child repos, ownership, local paths, Sergeant assignments, and resource allowlists.
- `resources.yaml` is the registry for logical Skills, MCPs, APIs, docs, and third-party resources. It contains no secrets.
- `maps/campaign-atlas.md` is generated from the manifests while preserving protected hand-written notes blocks.
- `maps/repos/*`, `.warroom/runs/*`, and `.warroom/dev/*` are local-only ignored paths.

## Normal Loop

```sh
npm run warroom -- doctor
npm run warroom -- sync --report
npm run warroom -- issue next
```

`issue next` prompts for a ready issue in an interactive terminal, creates and checks out a development branch, moves the selected source issue to `battlefield-active` on the Campaign Map, and runs the implementation handoff in the owning child repo through the foreground LLM adapter. For Codex this uses `codex exec`, so the command returns after the adapter exits and no background cloud or TUI session remains. After a successful interactive adapter run, War Room asks whether to run `warroom pr create` from that same implementation checkout. Same-repo issues use `gh issue develop --checkout`; ally issues with mapped `Owner repo` triage notes keep the issue in the ally repo but create a GitHub-linked branch in the owner repo with `createLinkedBranch` and store the source issue in branch metadata. The mapped checkout must be clean before launch. From a mapped child repo checkout, it lists only that repo's ready issues by default; use `--all` to show the cross-repo queue. The command ends with an explicit `Outcome:` line for completed, adapter-failed, dry-run, blocked, and not-started cases. Use `--dry-run` to preview without creating the branch, launching, or moving Campaign Map status.
Use `warroom commit create --repo <id> --validate "<command>" --write-artifact` before a final commit when the repo needs a deterministic change summary, validation record, and local audit bundle.
Use `warroom maps assign --repo <id>` with framework/domain/resource flags for reviewed specialist-context edits; pass `--write` only after checking the dry-run messages.

## Campaign Map

The Campaign Map is a GitHub Project board. Current project statuses are `needs-triage`, `ready-to-engage`, `battlefield-active`, `skirmish`, `blockaded`, and `victory`. Provision it with `warroom project create` (or wire an existing board with `warroom project link --project <number>`); both write `campaign_owner`/`campaign_project_number` into `repos.yaml`. `warroom setup` also offers this when `repos.yaml` has no `campaign_project_number`.

Use `warroom campaign status-check` to validate board status options and `warroom campaign status --issue owner/repo#number --status <status> --confirm` to move an issue deliberately. Any status change adds the issue to the Campaign Map board if it is not already there. The Campaign Map Status field is the single source of truth for an issue's phase — no duplicate workflow labels are written on the repo.

Workflow commands also understand the Campaign Map:

- `warroom issue next` reads `ready-to-engage` project items, then can select one for implementation, create the development branch, and move the source issue to `battlefield-active` on the Campaign Map. From a mapped repo checkout, the scoped list also includes ally issues whose triage notes name that repo as `Owner repo`.
- `warroom issue create` runs a PM-style interactive adapter session, previews the structured issue draft, then creates the GitHub issue only after confirmation. Confirmed creates add the issue to the Campaign Map with status `needs-triage`, apply ally labels for ally issue repos, best-effort set the issue type, and then ask whether to start `warroom issue triage` for the new issue.
- `warroom issue triage` reads `needs-triage` project items and can select one interactively to launch a planning-only triage handoff in the Codex TUI. The handoff may post triage notes back to the GitHub issue, but must not implement code. Ally issue repos are resolved through `allies.yaml` when they are not mapped in `repos.yaml`.
- Interactive `warroom issue triage` moves work to `ready-to-engage` only after the session exits successfully and War Room verifies a new `## War Room triage notes` issue comment with `Ready for ready-to-engage: yes`. Direct handoffs require `--mark-ready --confirm-status` for the same closeout.
- `warroom issue next --issue ... --confirm-status` moves work to `battlefield-active`.
- `warroom pr create --confirm --confirm-status` moves work to `skirmish` after publishing the PR.
- `warroom pr review --confirm-status` moves the linked issue to `skirmish`, using explicit `--issue`, queue context, branch metadata, or the PR body closing line.
- `warroom pr merge --confirm-status` moves the linked issue to `victory`, using explicit `--issue`, branch metadata, or the PR body closing line.

`warroom pr create` previews the PR title/body and branch push by default. Unless title/body flags are supplied, the preview asks the LLM adapter to draft PR text from the actual branch commits and diff. Large diffs are summarized through multiple adapter chunk calls before the final title/body prompt, so the final prompt has full diff coverage without a clipped patch. It falls back to a local issue/commit summary if adapter drafting fails. In an interactive terminal, the preflight asks whether to continue; a yes answer reuses the previewed PR text and runs the same confirmed path as `--confirm`. After a successful interactive create, War Room asks whether to run `warroom pr review` for the new PR. Successful confirmed creation prints the GitHub PR URL as the final line, while blocked or declined runs print a clear not-created outcome.

`warroom pr review` without `--pr` lists open PRs linked from issues in the `battlefield-active` and `skirmish` columns, ordered by latest PR update. If that queue is empty and the command is run inside a mapped child repo branch with a single open PR, it falls back to that current-branch PR. In an interactive terminal it confirms the single detected PR or asks for a numbered selection, then launches the selected review handoff as if `--pr <owner/repo#number> --launch` was passed. When launched from the `pr create` follow-up prompt, it first waits for CodeRabbit feedback on the initial PR commit and exits complete without an adapter run if no outstanding comments remain. Non-interactive current-branch fallback runs print the PR review preflight. Review and merge resolve the source issue from queue context, branch metadata, or the PR body closing line when `--issue` is omitted, so ally issues keep their client-facing home while implementation PRs live in mapped product repos. `warroom pr review --pr ... --launch` sends a fixed GitHub/CodeRabbit handoff that tells the adapter to analyze the latest CodeRabbit feedback on the latest PR commit, optionally mark each comment with eyes while working, reply to the review thread with `Change made:` or `Skipped:`, and commit the changes. The handoff includes the current CodeRabbit review-thread IDs and the GitHub thread-reply mutation to avoid relying on CodeRabbit auto-resolution alone. The eyes reaction is best-effort and must not block code fixes; final thread replies are required. The review loop uses the same foreground terminal adapter as other handoffs so it has local GitHub/CodeRabbit app access, `gh` auth, and a working PR branch remote. The CLI owns the loop: it logs each stage, waits for a new PR commit, waits for CodeRabbit to appear and settle on that commit, checks current unresolved CodeRabbit comments, verifies final replies on handled CodeRabbit threads, and relaunches the adapter while CodeRabbit feedback remains. Direct PR review handoffs end with an `Outcome:` line for completed, preflight-only, or blocked loops.
The loop defaults can be tuned with `WARROOM_PR_REVIEW_MAX_LOOPS`, `WARROOM_PR_REVIEW_COMMIT_TIMEOUT_MS`, `WARROOM_PR_REVIEW_CODERABBIT_TIMEOUT_MS`, `WARROOM_PR_REVIEW_CODERABBIT_SETTLE_MS`, and `WARROOM_PR_REVIEW_POLL_MS`.

`warroom pr merge` handoffs include merge state, GitHub mergeability, review decision, requested reviewers, unresolved review threads, draft state, status checks, readiness blocker explanations, the repo-specific merge gate decisions from `merge.playwright`, `merge.bump`, and `merge.changelog`, and a generated victory summary. If `--pr` is omitted, War Room infers the PR from the current mapped child repo branch by finding the single open GitHub PR for that branch. In an interactive terminal without `--confirm`, War Room prints the preflight and asks whether to continue into the confirmed merge path. On a blocked preflight, type `skip` to allow unresolved review threads if no other blockers remain. On a clear preflight, type `skip` to continue while skipping the demo Playwright gate. `--confirm`, a yes answer, or either interactive skip choice rechecks merge readiness. For repos with `merge.playwright: true`, War Room then checks whether `https://api.local.flopay.com/v1/health` is already running; if so War Room reuses that backend and leaves it running. If not, War Room starts the mapped backend with `npm run start:api`, waits for the health endpoint, runs the full demo `npm run test:e2e` Playwright suite from the mapped demo repo, and stops only the backend process it started. The confirmed run prints backend readiness progress and streams the Playwright command output so the terminal shows test progress and results unless the clear-preflight skip choice was used. Those repos are not merged unless all demo e2e tests pass or the clear-preflight skip choice was used. Repos with `merge.playwright: false` skip the backend/demo Playwright gate. If pre-merge gates pass, repos with `merge.bump` enabled ask `Should we bump the version number? [PATCH|minor|major|skip]` in interactive terminals; Enter runs the configured bump command with `patch`, commits the changed files to the PR branch, and pushes them before merge. Non-interactive runs skip the bump unless `--bump-version patch`, `--bump-version minor`, or `--bump-version major` is passed. War Room then runs `gh pr merge --squash --delete-branch`. For repos with `merge.changelog` enabled, War Room then asks `Run the public changelog update now ...? [y/N]` in interactive terminals; non-interactive runs require `--confirm-changelog`. Confirmed changelog updates wait for GitHub Actions on the base branch after merge, pull the latest base branch release/version files, use the foreground LLM adapter to update only the configured changelog target, and push a `[skip-ci]` changelog commit to the base branch. OpenChangelog targets create exactly one new public Markdown release note under the configured folder, usually `release-notes/`; legacy targets update `CHANGELOG.md`. Once the PR is merged and closeout gates are clear, War Room posts a short victory update to the linked issue unless `--no-issue-comment` is used. After a successful interactive confirmed merge, War Room prompts for the fuller victory summary posting and then prompts for local checkout cleanup, which switches to the PR base branch and pulls it with `git pull --ff-only`. Pass `--post-summary --confirm-summary` or `--cleanup-local --confirm-cleanup` to perform those follow-up actions without prompting. `--confirm-status` moves the linked issue to `victory` only when the merge-readiness preflight is clear.

The backend readiness probe uses a direct GET with a short per-request timeout. Local HTTPS certificates are accepted by default for `localhost`, loopback, and `*.local.flopay.com`; set `WARROOM_MERGE_BACKEND_STRICT_TLS=true` to require a trusted certificate, or tune the per-probe timeout with `WARROOM_MERGE_BACKEND_READY_PROBE_TIMEOUT_MS`. When the demo e2e run targets a local HTTPS backend, War Room passes `NODE_OPTIONS=--use-system-ca` to the Playwright process so the demo web server can trust locally installed development certificates. Set `WARROOM_MERGE_DEMO_USE_SYSTEM_CA=false` to disable that flag.

The version bump gate runs after Playwright and before PR merge: interactive runs ask `[PATCH|minor|major|skip]`, with `patch` selected by pressing Enter, and non-interactive runs use `--bump-version patch|minor|major|skip`.

The changelog gate is opt-in at merge closeout: interactive runs ask `[y/N]`, and non-interactive runs require `--confirm-changelog`. Once confirmed, it waits for GitHub Actions using `gh run list` on the PR base branch after merge, ignoring Dependabot update runs that do not gate release readiness. Tune it with `WARROOM_MERGE_CHANGELOG_ACTIONS_TIMEOUT_MS`, `WARROOM_MERGE_CHANGELOG_ACTIONS_POLL_MS`, and `WARROOM_MERGE_CHANGELOG_ACTIONS_SETTLE_MS` when release workflows need more time to publish version files. If the PR already merged but changelog closeout needs to be retried, run `warroom pr merge --pr <owner/repo#number> --resume-changelog --confirm --confirm-changelog`.

`warroom commit create` summarizes changed files, runs requested validation commands from the owning child repo, and writes ignored run artifacts when `--write-artifact` is present. From War Room root, it can infer the target repo from the single active mapped development branch written by `issue next`, including ally-source work implemented in a mapped product repo; pass `--repo <id>` if more than one child repo could match. Interactive runs print the dry run first and then ask before committing and pushing. Confirmed commits post a compact commit progress comment to the linked source issue unless `--no-issue-comment` is used, then interactive runs ask whether to run `warroom pr create` next. Non-interactive commits still require `--confirm`. If unstaged files are present, the interactive prompt explicitly confirms `git add -A`. Confirmed commits push to the current upstream, or to `origin` with upstream setup when the branch has no upstream. Use `--no-push` only for intentional local-only commits.

## Recovery

`warroom abort --print-recovery` is the first command to run when a multi-repo operation becomes unclear. It prints repo state and recovery commands without mutation. `--stash --confirm` is the preferred mutation when work should be preserved; `--danger-reset --confirm-danger "discard local work"` exists only as a last-resort local discard path.
